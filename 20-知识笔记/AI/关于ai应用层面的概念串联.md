## 📖 详细内容

### LLM
首先 LLM 是 Large Language Model ，也就是大语言模型，简称大模型，它的机制简单来说就是==根据你输入进去的话，它预测下一个词，然后再把输出的词和开始输入的作为新的输入，继续预测下一个词==。这就是为什么大模型是一个一个字进行输出。然而 LLM 实质上是以 Transerme 为基础的一个大函数，实质上输入的和输出的都是数字。所以就有了 tokenizer 。
[[LLM相关知识的简单理解]]
### token
tokenizer 可以理解为用户和 LLM 中间的一层，它负责编码和解码。编码需要切分，切分出来的一个个部分将会映射为一个数字，而这个个数字所代表的文本就称为一个 token。所以 **==token 是大模型理解的基本单位==**。而一个 token 大概等于 0.75 个英文，或者 1.5 个汉字。
[[token以及tokennizer的切分算法和机制]]
### context
那么为什么每次对话，LLM 可以记住你之前的问题和答案呢，其实每次你发送新的问题给 LLM 都会带上之前的一些对话，那么包含这些对话以及工具信息的整体就是 context ，中文名为上下文，可以理解为==**大模型的一个临时记忆体**==，那么就有了 context window 即上下文窗口这个概念，就是每次能够保存的最大上下文是多少，一般以 token 为单位，比如说 gpt 的 context window 为 100 万 token 。

### prompt
每个问出去的问题就是所谓的 prompt 中文叫提示词，==提示词的质量通常会决定大模型回答的质量==，一般提示词分为两类
- 系统提示词 —— system prompt，这个是开发者再后台进行配置的，给这个大模型一个身份，和一些规则。比如：你是一个医生，当病人问你一些和疾病相关的问题，你不仅要给它解释，还得带上免责说明，让它去具体的医院去看病。
- 用户提示词 —— user prompt 这个是用户输入的具体的问题，比如说 3+8 等于几。
### tool
正如刚刚所说，大模型实质上是一个函数，输入一些字符，它给出预测的输出，但是并不能真正的帮你做一些事情，所以就有人想能不能让大模型能够调用一些工具，比如说我问他天气，他能够用工具去查，而不是说知识库没有今天的天气信息之类的不痛不痒的话，由此诞生了 tool ，而怎么让大模型调用 tool ，这整个其实是一个具体的工作流程，我们引入 4 个角色，用户，平台，大模型和 tool ，那么简单来说运作模式大概是这样：
1. 用户发送问题：“帮我查询一下今天的天气” 给平台，而平台接入了各种 tool ，他会把 tool 的基本信息，比如 tool 的名称，需要那些参数，可以做什么功能以及用户的问题一起发送给大模型。
2. 大模型思考决策发现需要调用天气 tool 才能拿到结论，由此它发送调用该 tool 的格式和入参给平台，平台进行调用，调用结果再次返回给大模型，大模型判断是否有了用户想要的答案，再次返回给用户。

### mcp
而刚刚的 tool 有一个很大的缺陷，就是每个不同的大模型有不同的规范，导致 tool 开发者需要开发很多不同套的 tool 工具以供接入不同的平台，==而 mcp 的出现就是统一接入协议==，只要平台是支持 mcp 的，那么只需要开发一次tool工具就可以接入所有平台
[[MCP 原理]]
### agent
而 agent 其实再刚刚的 tool 流程中就已经可以看出雏形了，==agent 就是一个可以自主规划并调用工具处理的集成体==，比如查询天气，它可以自主规划要调用那些工具得到什么结果才是用户最终需要的，先调用定位工具定位用户再哪个城市，再调用时间工具确定日期，最后根据整个两个返回的结果去调用查询天气的工具。整体是一个 cycle，具体模式可以见下面整个笔记。

[[agent原理以及模式]]

### skill
当你做一些重复工作的时候，你希望它能够判断你做了重复工作，并且知道你一般的输出格式，而不是每次都在prompt把整套说明写进去，这时候就要用的skill，==它是agent的使用说明书==。比如说你要写论文，当设计到某些关键词的时候它会读取整个skill，然后按照里面说的规则，比如先调用什么工具创建文档，然后填充什么内容，格式是怎么样的。这样既可以不用每次都写重复的东西到prompt，又可以节省token
